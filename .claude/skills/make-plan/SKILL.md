# Make Plan

計画書（`documents/plan/<topic>.md`）を作成する。

**plan mode は使わない。** plan mode はユーザーに内容が見えないため。

## 引数

| パターン | 動作 |
|---------|------|
| `make-plan <topic>` | そのトピックの計画書を作成 |
| `make-plan <topic1> <topic2> ...` | 各トピックを別々の計画書として並列作成（1 topic = 1 sub-task） |
| `make-plan`（引数なし） | AskUserQuestion で何を作るか質問する |

## 手順（厳守）

**順番が重要。** Claude Code は「調査して詳細化してから計画書を作る」傾向があるが、それは逆。

### Phase 1: ファイル作成（即座に）

トピックを受け取ったら、**まず** `documents/plan/<topic>.md` にファイルを作成する。
この時点では骨格だけでよい。

```markdown
# <トピック名>

## 背景

<ユーザーの要求・指摘をそのまま記載>

## 調査中

- [ ] <調査項目1>
- [ ] <調査項目2>

## 方針

（調査後に記載）

## Spec・テストへの影響

（調査後に記載）

## 実行順序

（調査後に記載）

## 確認項目

（調査後に記載）
```

### Phase 2: 調査 & 詳細化（計画書を逐次更新）

1. 調査が必要な項目を洗い出す
2. **独立した調査は sub-task で並列実行する**（Task ツール、subagent_type: "Explore" or "general-purpose"）
3. 調査結果が返ってくるたびに **計画書ファイルを更新する**（Edit ツール）
4. 「調査中」セクションのチェックボックスを埋めていく
5. **Spec・テストへの影響を調査する**:
   - 方針で変更・新規作成するコードファイルを特定したら、対応する spec（`documents/spec/`）とテスト（`tests/`, インラインテスト）を確認する
   - 新規ファイル → spec 新規作成が必要か、テストが必要か
   - 既存ファイル変更 → 既存 spec の更新箇所、既存テストの追加・修正箇所を特定する
   - テスト困難な部分（OS 依存 API 等）はテスト方針（何をテストし、何をテストしないか）を明記する

### Phase 3: 曖昧な点の解消

調査の過程で曖昧な点が見つかったら:

1. **AskUserQuestion で質問する**（チャットのテキストではなく、ツールを使う）
2. 回答を受けたら計画書に反映する
3. 調査・詳細化の続きに取り組む

### Phase 4: 計画書の完成

全調査が完了したら:

1. 「調査中」セクションを削除し、結果を本文に統合する
2. 方針・実行順序・確認項目を記載する
3. **「Spec・テストへの影響」セクションを記載する**:
   - 新規作成が必要な spec・テストファイル一覧
   - 更新が必要な既存 spec・テストファイルと変更箇所
   - テスト方針（何をテストし、何をテストしないか。OS 依存等でテスト困難な部分の扱い）
   - 実行順序にも spec・テストの作業ステップを含める
4. 計画書の完成をユーザーに報告する

## 複数トピックの並列作成

トピックが複数の場合、各トピックを別々の sub-task で並列作成する。

```
各 Task 設定:
- subagent_type: "general-purpose"
- model: "sonnet"
- prompt: 下記テンプレートを使用
```

### sub-task プロンプトテンプレート

```
計画書作成タスク

## トピック
{topic}

## ユーザーの要求
{ユーザーが言ったこと}

## 手順
1. `documents/plan/{topic}.md` を Write ツールで作成する（骨格）
2. 関連するコード・spec・テストを調査する
3. 調査結果で計画書を Edit ツールで更新する
4. 方針・実行順序・確認項目を埋める
5. 変更・新規作成するコードに対応する spec とテストの影響を調査し「Spec・テストへの影響」セクションを埋める

## 調査対象のヒント
- コード: `code/frontend/`, `code/tauri/`
- テスト: `tests/integration/`, `code/tauri/src/timer.rs`（インラインテスト）
- Spec: `documents/spec/`
- 既存計画: `documents/plan/`
- 参考: `documents/reference/`

## 戻り値
作成した計画書の絶対パスと、ユーザーに確認すべき曖昧な点のリスト。
```

## 禁止事項

- **EnterPlanMode を使わない** — ユーザーに内容が見えないため
- **調査を全て終えてから計画書を作らない** — ファイル作成が先
- **コード・テスト・spec を修正しない** — 計画書の作成のみ
- **曖昧な点を自分で判断しない** — ユーザーに質問する
