# Tests to Code

テストを真としてコードを評価・修正するスキル。

## クイックリファレンス

1. **Pre**: git commit & tag（ロールバックポイント作成）
2. **対象選定**: テストファイルを収集（`tests/integration/`, `code/tauri/src/`（インラインテスト））
3. **静的分析**: サブエージェント並列でテスト↔コード整合性チェック（クラスC）
4. **実行診断**: サブエージェント並列でテスト実行・原因分類（クラスA/B）
5. **サマリー**: `.code-from-tests/summary.md` に出力
6. **ユーザー確認**: 許可取得
7. **修正適用**: 許可後にクラスA/C1を修正、クラスB/C2を報告
8. **Post**: git commit & tag（完了記録）

## 概要

| 項目 | 内容 |
|------|------|
| 入力 | テストディレクトリ or テストファイル |
| 出力 | `code/.code-from-tests/` に診断結果 |
| 原則 | testsを真とする（codeに迎合的にtestsを修正しない） |
| テンプレート | `templates/diagnosis.md` |

## 変更管理方針

**gitを主体とした変更管理を行う。**

- `git tag`でロールバックポイントを作成
- 診断結果（What/Why/How）は`diagnosis/`に記録
- 実際の修正はEditツールで直接適用
- 問題発生時は`git revert`や`git reset`でロールバック

## 問題の分類

### クラスC: 整合性問題（静的分析で検出）

**テストとソースコードの比較**で検出する問題。テスト実行は不要。

| レベル | 問題 | 説明 | アクション |
|--------|------|------|-----------|
| C1 | 自己充足テスト | テスト内にソースロジックを再実装している | テスト修正（Class Aと同様） |
| C2 | IF不整合 | テスト対象のインターフェースと実装が乖離 | コード修正（Class Bと同様） |

**C1判定の詳細:**
以下は **C1に該当しない**（除外）:
- モックオブジェクト・スタブの定義
- テストデータ・フィクスチャの定義
- テストヘルパー関数

以下は **C1に該当する**:
- ソースコードと同じ計算ロジックの再実装
- ソースの内部アルゴリズムの複製
- 期待値をロジックで動的生成（ソースと同じ計算）

### クラスA: テストの問題（実行診断で検出）

**テストファイル内**の問題。

| レベル | 問題 | 例 |
|--------|------|-----|
| A1 | import/use文 | `use wrong_crate` → `use correct_crate` |
| A2 | テスト構造 | async/await漏れ、重複定義、不正なテストデータ |
| A3 | 環境 | Rustツールチェイン、Node.jsバージョン |
| A4 | セットアップ | テストセットアップの不備 |
| A5 | 順序依存 | 他テストの副作用に依存 |
| A6 | モック/データ | モック不備、テストデータの形式エラー |

**具体例:**
- テストデータに不正なULID `L` が含まれる → A6
- テストのモックレスポンスが古い形式 → A6
- テストが無効なULIDで404を期待、コードが正しく400を返す → A6（コードは正常動作）

### クラスB: コードの問題（実行診断で検出）

**コードファイル**の問題。

**やってはいけないこと:**
- specを見る
- テストで帳尻を合わせる
- どちらが正しいか判断する

事実のみ記録する。

| レベル | 問題 | 例 |
|--------|------|-----|
| B1 | モジュール形式 | crateの公開設定（pub）不足 |
| B2 | アクセスレベル | pub(crate)/privateでテスト困難 |
| B3 | 暗黙依存 | グローバル変数経由の依存 |
| B4 | 本質的不一致 | **有効な入力**に対して期待値 ≠ 実際値 |

**具体例:**
- コードが正しい入力に対して間違った値を返す → B4
- コードがuse不可（pub不足、crate未公開） → B1

## 迎合的修正の禁止

```
テスト失敗: 「期待値は10、実際は20」
    ↓
❌ 迎合的: 期待値を20に書き換える
✅ 正しい: codeを直して10を返すようにする
```

## ワークフロー

### Step 0: Pre - git commit & tag

```bash
git add -A
git diff --cached --quiet || git commit -m "pre: tests-to-code checkpoint"
git tag tests-to-code-pre-$(date +%Y%m%d-%H%M%S)
```

### Step 1: 対象ファイル特定

- `tests/integration/` および `code/tauri/src/`（インラインテスト）を対象とする
- `.tests-from-spec/generated/` などの作業ディレクトリは対象外
- ユーザー指定があればそれに従う

### Step 2a: 静的分析フェーズ（サブエージェント並列）

対象の全テストファイルに対して、テストとソースコードを読み比べてクラスCの問題を検出する。

1. テストファイルをReadで読み込む
2. import文からテスト対象のソースコードを特定し、Readで読む
3. クラスCチェック:
   - C1: テスト内にソースロジックの再実装がないか？
   - C2: テストが期待するIFとソースの公開IFが一致しているか？
4. 問題があれば `diagnosis/{相対パス}.md` に出力

### Step 2b: 実行診断フェーズ（サブエージェント並列）

対象の全テストファイルに対してサブエージェントを並列起動し、テストを実行する。

**サブエージェントへの指示**: テンプレート `templates/diagnosis.md` の `{変数}` を埋めて渡す。サブエージェントはテンプレートをfillして `diagnosis/{相対パス}.md` に出力する。

**重要:**
- 「失敗しているファイルだけ」に絞らない。全対象ファイルを診断する
- テスト成功でも前提チェックが外れていればB1として報告する

### Step 3: サマリー生成 + ユーザー報告

`code/.code-from-tests/summary.md` を生成し、ユーザーに報告して許可を得る。

### Step 4: 修正適用（許可後）

1. `diagnosis/`から修正内容（What/Why/How）を確認
2. Editツールで修正:
   - クラスA・C1: テストファイルを修正
   - クラスC2: コードファイルを修正
3. 修正後、テストを再実行して確認

クラスBの問題は修正せず、報告のみ。

### Step 5: Post - git commit & tag

```bash
git add -A
git commit -m "tests-to-code: {修正件数} fixes, {報告件数} reports"
git tag tests-to-code-post-$(date +%Y%m%d-%H%M%S)
```

## 出力構造

```
code/
├── .code-from-tests/
│   ├── summary.md
│   └── diagnosis/
│       ├── auth/
│       │   └── session-sync.md
│       └── ...
```

## summary.md フォーマット

```markdown
# Tests to Code Summary

実行日: {YYYY-MM-DD}

## Diagnosis一覧

| Test | Source | Check | Class |
|------|--------|-------|-------|
| {テストファイルパス} | {ソースコードパス} | {pass/fail} | {-/A1-A6/B1-B4/C1-C2} |
```
