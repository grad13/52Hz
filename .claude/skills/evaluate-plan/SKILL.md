# Evaluate Plan

計画書（`documents/plan/*.md`）を作成・更新した直後に実行する。
計画書の品質を100点満点で自己評価し、不足点を特定・改善し、最終的に矛盾や漏れがないことを確認する。

ultrathink

## クイックリファレンス

1. **Phase 1**: 対象特定 + 全文読み込み + 関連情報収集
2. **Phase 2**: 10カテゴリ × 10点で採点 → 重要度分類
3. **Phase 3**: 8点未満のカテゴリを改善（計画書を直接更新）
4. **Phase 4**: 最終検証（内部一貫性 + 実態との整合性、最大3回）
5. **Phase 5**: 評価結果の報告

## 対象

`documents/plan/*.md`（README.md 除外、使用終了日が設定済みのものも除外）

## 実行モード

| モード | 引数 | 動作 |
|--------|------|------|
| 単体 | ファイル1つ | 直接評価（Phase 1〜5 をメインエージェントで実行） |
| 並列 | ファイル複数 or 引数なし | サブエージェント並列（1 plan = 1 agent） |

### 単体モード

```
evaluate-plan documents/plan/2026-02-23_topic.md
```

### 並列モード

```
evaluate-plan                                    # 全アクティブplan
evaluate-plan documents/plan/a.md documents/plan/b.md  # 複数指定
```

引数なしの場合:
```bash
# アクティブな計画書を収集（使用終了日が `-` のもの）
# README.md は除外
```

---

## Phase 1: 計画書の読み込みと現状把握

### Step 1-1: 対象の特定

- **単体モード**: 引数で指定された計画書
- **並列モード**: 引数で指定された複数ファイル、または全アクティブplan

### Step 1-2: 全文読み込みと関連情報収集

**単体モード**: メインエージェントが直接読み込む。
**並列モード**: 各サブエージェントが担当ファイルを読み込む。

確認対象:
- 計画書本体（`documents/plan/<topic>.md`）
- 計画書が参照しているドキュメント（`documents/spec/`, `documents/reference/`）
- 計画書が参照している外部資料（`externals/`）
- 計画書が修正対象とするコード（`code/`）
- 計画書が修正対象とするテスト（`tests/`）

---

## Phase 2: 100点満点の自己評価

以下の10カテゴリ × 10点満点で評価する。各カテゴリの採点基準に従い、厳密に採点すること。

### 評価カテゴリ

| # | カテゴリ | 配点 | 評価観点 |
|---|---------|------|---------|
| 1 | **基本構造** | 10点 | 必要なセクション（背景・方針・確認項目・実行順序）が揃っているか |
| 2 | **問題の理解** | 10点 | ユーザーの指摘を原文引用し、問題を正確に把握しているか。読み手が迷わないか |
| 3 | **情報源と仕様の把握** | 10点 | exemplar・spec・既存コード・テストを確認してパスを明記し、仕様を自分の言葉で正確に説明しているか |
| 4 | **Spec 影響分析** | 10点 | 変更対象コードに対応する spec を特定し、既存 spec との整合性確認・spec の更新/作成計画が含まれているか |
| 5 | **修正・実装方針** | 10点 | ファイル・行番号・変更内容が具体的か。「何をどう変える」が明確か |
| 6 | **エッジケース・異常系** | 10点 | 境界条件・異常系・副作用が考慮されているか |
| 7 | **テスト・検証計画** | 10点 | テスト方針・検証手順が記載されているか。変更後の確認方法が明確か |
| 8 | **既存コードとの整合性** | 10点 | 既存のコード・テスト・spec との整合性が確認されているか（3側面一致） |
| 9 | **確認項目** | 10点 | ユーザーに確認すべき点がチェックリスト形式で明記されているか |
| 10 | **実行順序** | 10点 | 依存関係・フェーズ分割が適切か。手順の順番に論理的根拠があるか |

### #4 Spec 影響分析の評価観点

| 観点 | 10点（完璧） | 5点（不足あり） | 0点（未記述） |
|------|------------|---------------|------------|
| 影響 spec の特定 | 変更対象コードに対応する spec を全て列挙し、パスを明記 | 一部の spec しか確認していない、または存在確認が曖昧 | spec への言及がない |
| 既存 spec との整合性 | 提案する変更が既存 spec の Contract/Logic/State/SideEffects と矛盾しないことを確認済み | 確認が部分的（Contract だけ見ている等） | 整合性チェックなし |
| Spec 更新/作成計画 | 変更後に必要な spec 更新（既存 spec の修正・新規 spec 作成）が実行順序に組み込まれている | 「spec も更新する」と書いてあるが具体性がない | spec 更新への言及なし |
| spec がない場合の対応 | 対応 spec が存在しない場合、新規作成するか判断と理由が記載されている | 言及はあるが判断基準が不明確 | 未検討 |

注: 変更対象が spec に影響しない計画書（ドキュメント整理、ツール変更等）では N/A として扱う。

### CLAUDE.md 計画書規約との対応

| CLAUDE.md 規約 | 対応カテゴリ |
|---------------|------------|
| 問題の理解: ユーザーの指摘を原文引用 | #2 問題の理解 |
| 確認した exemplar: ファイルパスを明記 | #3 情報源と仕様の把握 |
| 理解した仕様: 自分の言葉で説明 | #3 情報源と仕様の把握 |
| 修正・実装方針: 具体的な変更内容 | #5 修正・実装方針 |
| Spec 影響: 影響 spec 特定・更新計画 | #4 Spec 影響分析 |
| 確認項目: チェックリスト形式 | #9 確認項目 |

### 採点基準（各カテゴリ共通）

| 点数 | 基準 |
|------|------|
| 9-10 | 完璧。追加すべき内容がない |
| 7-8 | 良好。軽微な追記で完成する |
| 5-6 | 普通。重要な情報が一部欠けている |
| 3-4 | 不十分。大幅な追記が必要 |
| 1-2 | 骨格のみ。ほぼ未記述 |
| 0 | セクション自体が存在しない |

**甘い採点は禁止。** 実装に必要な情報が欠けていれば容赦なく減点する。
評価は計画書の「実装着手に十分な情報があるか」を基準とする。

### Step 2-1: 採点の実施

各カテゴリを採点し、結果を整理する。

### Step 2-2: 重要度による分類

8点未満のカテゴリを改善対象として抽出し、優先度を付ける:

- **Critical（0-4点）**: 計画書として機能しない。即座に対応が必要
- **Major（5-6点）**: 実装着手に支障あり。改善が強く推奨される
- **Minor（7点）**: 品質向上のため推奨。なくても実装は可能

---

## Phase 3: 改善タスクリストの作成と計画書の更新

### Step 3-1: タスクリストの作成

Phase 2 の評価結果から改善タスクリストを作成する。

タスクの粒度:
- 1タスク = 計画書の1セクションの追加・改善
- 具体的に何を書くかまで記述する（例: 「エッジケースセクションに境界条件を追記」）

### Step 3-2: タスクの実行（計画書の更新）

タスクリストに従い、計画書を更新する。

更新ルール:
- **既存の記述は尊重する**: 矛盾がない限り既存内容を変更しない
- **セクションの追加**: 不足しているセクションは計画書のフォーマットに合わせて追加
- **具体性を重視**: 抽象的な記述より、ファイルパス・行番号・コード例を含む具体的な記述
- **実装コードとの整合性**: ソースコードが既に存在する場合、実装と一致させる

### Step 3-3: 各タスク完了後の確認

タスクを1つ完了するごとに:
1. 該当セクションを再読して自然な文章になっているか確認
2. 前後のセクションとの整合性を確認

---

## Phase 4: 最終検証（矛盾・収載漏れチェック）

計画書の更新が完了したら、全文を再読して以下を検証する。

### Step 4-1: 内部一貫性チェック

- 背景で述べた問題と、修正・実装方針が対応しているか
- 確認した情報源で参照したファイルが、方針に反映されているか
- 確認項目が、方針の各ステップに対応しているか
- 実行順序の依存関係に矛盾がないか

### Step 4-2: 実態との整合性チェック

- 計画書が参照しているファイルパスが実在するか
- 計画書が参照している行番号が正しいか（Readで確認）
- 修正方針が現在のコードの実態と一致しているか

### Step 4-3: 問題が見つかった場合

矛盾や漏れが見つかった場合:
1. 問題箇所を特定する
2. 計画書を修正する
3. 再度 Step 4-1 〜 4-2 を実行する

問題がなくなるまで繰り返す（最大3回）。

---

## Phase 5: 評価結果の報告

最終的な評価結果をユーザーに報告する。

報告フォーマット:

```
## 計画書評価レポート: <計画書名>

### 初回評価: XX/100点
### 改善後評価: YY/100点

### 採点詳細

| # | カテゴリ | 初回 | 改善後 | コメント |
|---|---------|------|--------|---------|
| 1 | 基本構造 | X/10 | Y/10 | ... |
| 2 | 問題の理解 | X/10 | Y/10 | ... |
| 3 | 情報源と仕様の把握 | X/10 | Y/10 | ... |
| 4 | Spec 影響分析 | X/10 | Y/10 | ... |
| ... | ... | ... | ... | ... |
| | **合計** | **XX/100** | **YY/100** | |

### 改善内容
- <改善1>
- <改善2>
- ...

### 最終検証結果
- 内部一貫性: pass / fail（詳細）
- 実態との整合性: pass / fail（詳細）
```

### 閾値

| 合計点 | 判定 |
|--------|------|
| 90+ | 実装着手可能 |
| 90未満 | 改善が必要（不足点を提示し、再改善を提案） |

100点は理想的な状態であり、90点以上であれば実装着手可能と判断する。
90点未満の場合は、実装着手前に計画書の改善が必要であることを明確に伝える。

## 計画書の種類による適用

全ての計画書が10カテゴリ全てに該当するわけではない。

| 計画書の種類 | 重要度が高いカテゴリ | 軽くてよいカテゴリ |
|------------|-------------------|------------------|
| バグ修正 | #2問題理解, #4Spec影響, #5方針, #6エッジケース | #1構造（軽量でよい） |
| 機能追加 | #3情報源仕様, #4Spec影響, #5方針, #7テスト計画 | #6エッジケース（段階的でよい） |
| リファクタリング | #4Spec影響, #8整合性, #10実行順序 | #2問題理解（動機は簡潔でよい） |
| 提言・検討 | #2問題理解, #9確認項目 | #4Spec影響, #5方針, #7テスト計画（未確定でよい） |

該当しないカテゴリは「N/A」として採点から除外し、残りのカテゴリで按分する。

例: 提言書で #5, #7 が N/A の場合 → 8カテゴリ × 10点 = 80点満点、72点以上で合格（90%）。

全サブエージェントを**1つのメッセージで同時に発行**して並列実行する。

```

- model: "sonnet"
```

```
## 禁止事項

- 甘い採点
```

## 注意事項

- **甘い採点禁止**: 実装に必要な情報が欠けていれば容赦なく減点する
- **改善は計画書のみ**: コード・テスト・spec は修正しない
- **実態確認必須**: Phase 4 でファイルパス・行番号を実際に確認する
- **N/A の適切な使用**: 該当しないカテゴリは無理に採点せず N/A とする
- **更新日の更新**: 計画書を修正した場合、frontmatter の更新日を今日の日付にする
- **並列モードのモデル**: サブエージェントは `sonnet` を使用（コスト効率）
